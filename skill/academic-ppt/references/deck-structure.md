# Deck Structure

## Default Page Order

Default thesis-defense sequence:

1. Cover
2. Table of contents
3. Research background and significance
4. Problem definition or research objective
5. Technical route or overall framework
6. Student work part 1
7. Student work part 2
8. Experimental setup or evaluation protocol
9. Core results
10. Innovation summary
11. Limitations and future work
12. Closing or thanks

## Mandatory Page Types By Scenario

### General Thesis Defense

Must include:
- cover
- research problem
- method/framework
- evidence/results
- innovation
- conclusion

### Experimental Thesis

Must include:
- dataset or experimental object
- evaluation metrics
- comparison baseline or control setup
- result interpretation

### Engineering Or System Thesis

Must include:
- system architecture
- key module design
- implementation or deployment evidence
- functional or performance verification

## Slide-Level Rules

- One main claim per slide
- One complex chart per slide unless multiple visuals support the same conclusion
- If a page needs more than one minute of explanation just to decode the visual, simplify or split it
- Put innovation evidence near the relevant content, not only at the end
- Most content slides should stay within 2 to 3 visible evidence bullets
- Method and result pages should reserve substantial space for a figure, chart, or architecture visual

## Page-Type Layout Budget

### Cover

- Title first
- Metadata second
- Branding third

### Background / Problem

- top: one strong judgment sentence
- body: 2 to 3 short supporting points
- optional side panel only if it clarifies the scenario

### Technical Route / Method

- text side explains the logic
- visual side explains the structure
- do not duplicate the same information in both areas

### Experiment / Results

- top band for metric cues or key comparisons
- lower area split into interpretation and chart
- chart captions should explain what changed and why it matters

### Innovation

- maximum 3 separated claims
- each claim should stay compact enough to scan in one glance

### Limitations

- use a dual-column split
- left: current limitations
- right: future work

## Title Rules

Prefer titles that summarize the conclusion.

Good:
- "The proposed pipeline reduces annotation cost while preserving accuracy"
- "The redesigned module improves retrieval precision on three datasets"

Weak:
- "Method"
- "Experiment"
- "Results"
